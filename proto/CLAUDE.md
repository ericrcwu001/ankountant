# proto

Read before editing proto.

The `.proto` files here are the backend RPC contract. Editing one regenerates
the Rust, Python, and TypeScript bindings the three layers use to talk to each
other, so it needs a full `just check` — not just `cargo check`.

## What lives here

- `proto/anki/*.proto` — service and message definitions (one file per area:
  `decks.proto`, `notes.proto`, `scheduler.proto`, `i18n.proto`, etc.). Sources
  of truth for all generated bindings.
- `proto/anki/backend.proto` — the umbrella `Backend*Service` wiring.
- The codegen itself lives outside this tree, in `rslib/proto/` and
  `rslib/proto_gen/` (see cross-references below).

## Regeneration flow

Editing a `.proto` triggers, via `rslib/proto/build.rs`:

- Rust service traits + dispatch — included into `rslib/src/services.rs` from
  `OUT_DIR/backend.rs` (generated by `rslib/proto/rust.rs` and `rslib/build.rs`).
- Python snake_case wrappers — `out/pylib/anki/_backend_generated.py` (generated
  by `rslib/proto/python.rs`). `RustBackend` in `pylib/anki/_backend.py` extends
  these and bridges FFI via `_run_command(service, method, input)`.
- TypeScript async callers — `out/ts/lib/generated/backend.ts` (generated by
  `rslib/proto/typescript.rs`), imported elsewhere as `@generated/backend`. Each
  function calls `postProto()`, which POSTs to `/_anki/{method}`.

These outputs are transient build artifacts under `out/`. Never edit or commit
them.

## Gotchas

- Service/method indices must match across all three layers. `proto_gen` assigns
  them by service/method order in the descriptor pool; Python embeds them,
  TypeScript embeds method names, and the generated Rust `run_service_method()`
  dispatches on them. Stale codegen means Python/TS calls the wrong Rust method
  (hangs, crashes, or 404s) — this is why a full `just check` is mandatory.
- Index instability: adding, reordering, or removing a service or method shifts
  every index after it. Add new services/methods at the bottom of the list.
- Fourth consumer — iOS (no codegen): unlike Rust/Python/TS, there is no
  `swift.rs` generator. The Swift client's service/method indices are
  hand-maintained in `ios/Sources/AnkiBackend/AnkiBackend.swift` and must be
  re-derived from `out/pylib/anki/_backend_generated.py` after any add/reorder,
  or iOS dispatches to the wrong method. See `ios/CLAUDE.md`.
- Dual-service pattern: each area defines both `XxxService` (collection-facing,
  `&mut self`) and `BackendXxxService` (backend-facing, `&self`). Backend methods
  not explicitly listed delegate to the collection service; `proto_gen` panics if
  the pairing is malformed.
- Keep proto changes backward-compatible — old collections hold persisted data.

## Ankountant work

The #1 cross-cutting gotcha: a `.proto` edit is a contract change across Rust,
Python, and TypeScript. After editing, run `just check` so all three layers
regenerate in sync. The Rust trait impls these dispatch to live in
`rslib/src/backend/*.rs` (per area).
